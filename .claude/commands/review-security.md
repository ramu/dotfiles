# セキュリティレビューコマンド

指定されたコミット範囲のコード変更を、国際的なセキュリティ規格に基づいて分析し、脆弱性を検出します。問題が見つかった場合は修正タスクリストを生成し、ralph-loop で再修正できるようにします。

**準拠規格**:
- [OWASP Top 10 (2021)](https://owasp.org/Top10/)
- [CWE Top 25 Most Dangerous Software Weaknesses (2023)](https://cwe.mitre.org/top25/archive/2023/2023_top25_list.html)

## 引数の解析

`$ARGUMENT` をスペース区切りで解析:

| 引数 | 必須 | 説明 | デフォルト |
|------|------|------|-----------|
| 第1引数 | - | コミット範囲 | 自動検出 |
| 第2引数 | - | 修正タスクリストの出力パス | `~/tmp/claude-dev/{プロジェクト名}/review-security-fixes.md` |

### コミット範囲の自動検出

第1引数が省略された場合:
1. 現在のブランチとデフォルトブランチ（main/master）の分岐点を検出
2. 分岐点から HEAD までの全コミットを対象とする
3. デフォルトブランチ上にいる場合は直近 5 コミットを対象とする

## 実行フロー

### Step 1: 対象コミットの特定

1. コミット範囲を確定する（引数または自動検出）
2. `git log --oneline {範囲}` で対象コミット一覧を表示
3. `git diff {範囲}` で変更の全体差分を取得

### Step 2: 変更ファイルの読み込み

変更されたファイルの**現在の内容**を読み込む（差分だけでなく、変更後のファイル全体を確認することで、コンテキストを把握する）。

### Step 3: セキュリティレビュー観点のチェック

以下の観点で変更を分析する。各項目は OWASP Top 10 (2021) および CWE Top 25 (2023) に準拠している。

#### [OWASP A01] アクセス制御の不備 — Broken Access Control
> https://owasp.org/Top10/A01_2021-Broken_Access_Control/

- [CWE-200] 権限のないアクターへの機密情報の露出
- [CWE-284] 不適切なアクセス制御
- [CWE-285] 不適切な認可
- [CWE-352] クロスサイトリクエストフォージェリ（CSRF）
- [CWE-425] 強制ブラウジング / ダイレクトリクエスト
- [CWE-862] 認可チェックの欠如
- [CWE-863] 不正な認可

#### [OWASP A02] 暗号化の失敗 — Cryptographic Failures
> https://owasp.org/Top10/A02_2021-Cryptographic_Failures/

- [CWE-259] ハードコードされたパスワードの使用
- [CWE-327] 壊れたまたはリスクのある暗号アルゴリズムの使用
- [CWE-331] 不十分なエントロピー
- [CWE-798] ハードコードされた認証情報の使用
- 平文での機密データ送信・保存
- 弱い暗号アルゴリズム（MD5, SHA1, DES 等）の使用

#### [OWASP A03] インジェクション — Injection
> https://owasp.org/Top10/A03_2021-Injection/

- [CWE-20] 不適切な入力バリデーション
- [CWE-77] コマンドインジェクション
- [CWE-78] OS コマンドインジェクション
- [CWE-79] クロスサイトスクリプティング（XSS）
- [CWE-89] SQL インジェクション
- [CWE-94] コードインジェクション
- [CWE-116] 出力の不適切なエンコードまたはエスケープ
- [CWE-917] サーバーサイドテンプレートインジェクション（SSTI）

#### [OWASP A04] 安全でない設計 — Insecure Design
> https://owasp.org/Top10/A04_2021-Insecure_Design/

- [CWE-256] 保護されていない認証情報の保存
- [CWE-501] 信頼境界の違反
- [CWE-522] 十分に保護されていない認証情報
- ビジネスロジックの欠陥（レースコンディション、TOCTOU）
- 脅威モデリングの不足（重要な操作にセキュリティ制御がない）

#### [OWASP A05] セキュリティの設定ミス — Security Misconfiguration
> https://owasp.org/Top10/A05_2021-Security_Misconfiguration/

- [CWE-16] 環境設定の問題
- [CWE-611] XML 外部エンティティ（XXE）
- デフォルト認証情報の使用
- 不要な機能の有効化（デバッグモード、不要なポート）
- セキュリティヘッダーの欠如（CORS, CSP, HSTS 等）
- 過度に許可的な設定（パーミッション、ファイアウォールルール）

#### [OWASP A06] 脆弱で古いコンポーネント — Vulnerable and Outdated Components
> https://owasp.org/Top10/A06_2021-Vulnerable_and_Outdated_Components/

- 既知の脆弱性を持つライブラリ・フレームワークの使用
- バージョン固定されていない依存関係
- サポート終了（EOL）のコンポーネント

#### [OWASP A07] 識別と認証の失敗 — Identification and Authentication Failures
> https://owasp.org/Top10/A07_2021-Identification_and_Authentication_Failures/

- [CWE-287] 不適切な認証
- [CWE-306] 重要な機能に対する認証の欠如
- [CWE-798] ハードコードされた認証情報
- 弱いパスワードポリシーの許可
- セッション管理の不備（固定セッション、不適切な有効期限）
- ブルートフォース保護の欠如

#### [OWASP A08] ソフトウェアとデータの整合性の失敗 — Software and Data Integrity Failures
> https://owasp.org/Top10/A08_2021-Software_and_Data_Integrity_Failures/

- [CWE-502] 信頼できないデータのデシリアライゼーション
- [CWE-829] 信頼できない制御領域からの機能の組み込み
- 整合性チェックなしの自動更新
- CI/CD パイプラインの保護不足

#### [OWASP A09] セキュリティログとモニタリングの失敗 — Security Logging and Monitoring Failures
> https://owasp.org/Top10/A09_2021-Security_Logging_and_Monitoring_Failures/

- [CWE-117] ログへの不適切な出力の無害化
- [CWE-532] ログファイルへの機密情報の挿入
- 認証失敗・アクセス制御失敗のログ記録の欠如
- ログに含まれる機密情報（パスワード、トークン、個人情報）

#### [OWASP A10] サーバーサイドリクエストフォージェリ — SSRF
> https://owasp.org/Top10/A10_2021-Server-Side_Request_Forgery_(SSRF)/

- [CWE-918] サーバーサイドリクエストフォージェリ（SSRF）
- ユーザー入力由来の URL へのサーバーサイドリクエスト
- 内部リソースへのアクセス制御の欠如

#### CWE Top 25 追加項目（OWASP カテゴリ外）

以下は CWE Top 25 (2023) に含まれるが上記 OWASP カテゴリで十分にカバーされていない項目:

- [CWE-119] メモリバッファの範囲内での操作の不適切な制限
- [CWE-125] 境界外読み取り
- [CWE-190] 整数オーバーフローまたはラップアラウンド
- [CWE-269] 不適切な権限管理
- [CWE-276] 不適切なデフォルトパーミッション
- [CWE-416] 解放済みメモリの使用（Use After Free）
- [CWE-434] 危険なタイプのファイルの無制限アップロード
- [CWE-476] NULL ポインタ参照
- [CWE-787] 境界外書き込み

### Step 4: 結果の報告

レビュー結果を以下の形式で表示する:

```
═══ セキュリティレビュー結果 ═══

準拠規格: OWASP Top 10 (2021) / CWE Top 25 (2023)
対象: {コミット範囲} ({N} コミット, {M} ファイル変更)

## 問題なし ✓
- {チェック済みで問題のなかった項目と対応する規格識別子}

## 要修正（致命的）
- {ファイル}:{行} - [{識別子}] {問題の説明}

## 要修正（重要）
- {ファイル}:{行} - [{識別子}] {問題の説明}

## 提案（軽微）
- {ファイル}:{行} - [{識別子}] {改善の提案}

---
致命的: {N} / 重要: {N} / 軽微: {N}
```

### Step 5: 修正タスクリスト生成（問題がある場合）

「致命的」または「重要」の問題が 1 つ以上ある場合:

1. 修正タスクリストを生成する
2. 形式は decompose.md の出力と同じ Markdown チェックリスト
3. 各タスクには問題の場所（ファイル:行）、規格識別子、修正方針を記載

```markdown
# セキュリティレビュー修正タスク

> レビュー対象: {コミット範囲}
> 準拠規格: OWASP Top 10 (2021) / CWE Top 25 (2023)
> 生成日: {YYYY-MM-DD}

## タスク
- [ ] {ファイル}:{行} - [{識別子}] {具体的な修正内容}
...
```

4. 出力パスを表示し、ralph-loop での実行コマンド例を案内

### Step 6: 問題がない場合

全チェックで問題が見つからなかった場合:
- 「問題は検出されませんでした」と報告
- 修正タスクリストは生成しない

## ルール

- **変更のコンテキストを理解する**: 差分だけでなく、変更後のファイル全体を読む
- **プロジェクト規約を基準にする**: 一般的なベストプラクティスよりも、プロジェクトの既存パターンを優先する
- **誤検出を避ける**: 確信のない問題は「提案（軽微）」として報告し、致命的・重要には含めない
- **修正可能な問題のみ報告**: 設計方針への意見や好みの問題は報告しない
- **言語・フレームワークに応じた判断**: 使用言語やフレームワークが提供する安全機能を考慮し、過剰な指摘を避ける（例: ORM を使用している場合の SQL インジェクション指摘）
