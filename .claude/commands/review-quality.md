# コード品質レビューコマンド

指定されたコミット範囲のコード変更を、国際的なソフトウェア品質規格に基づいて分析し、品質上の問題を検出します。問題が見つかった場合は修正タスクリストを生成し、ralph-loop で再修正できるようにします。

**準拠規格**:
- [ISO/IEC 25010:2023 — システム及びソフトウェア品質モデル](https://www.iso.org/standard/78176.html)
- [ISO/IEC 5055:2021 — ソフトウェア品質の自動測定](https://www.iso.org/standard/80623.html)
- [CISQ Code Quality Standards](https://www.it-cisq.org/standards/code-quality-standards/)

## 引数の解析

`$ARGUMENT` をスペース区切りで解析:

| 引数 | 必須 | 説明 | デフォルト |
|------|------|------|-----------|
| 第1引数 | - | コミット範囲 | 自動検出 |
| 第2引数 | - | 修正タスクリストの出力パス | `~/tmp/claude-dev/{プロジェクト名}/review-quality-fixes.md` |

### コミット範囲の自動検出

第1引数が省略された場合:
1. 現在のブランチとデフォルトブランチ（main/master）の分岐点を検出
2. 分岐点から HEAD までの全コミットを対象とする
3. デフォルトブランチ上にいる場合は直近 5 コミットを対象とする

## 実行フロー

### Step 1: 対象コミットの特定

1. コミット範囲を確定する（引数または自動検出）
2. `git log --oneline {範囲}` で対象コミット一覧を表示
3. `git diff {範囲}` で変更の全体差分を取得

### Step 2: 変更ファイルの読み込み

変更されたファイルの**現在の内容**を読み込む（差分だけでなく、変更後のファイル全体を確認することで、コンテキストを把握する）。

### Step 3: 品質レビュー観点のチェック

以下の観点で変更を分析する。各項目は ISO/IEC 25010:2023 の8品質特性および ISO/IEC 5055:2021 の4品質指標に準拠している。

---

#### [25010-信頼性] 信頼性 — Reliability

> ソフトウェアが指定された条件下で、指定された期間、故障なく動作する能力

**[5055-REL] 信頼性の自動測定指標**

- [25010-信頼性: 成熟性] 通常運用時の障害回避
  - 明らかなバグ（off-by-one エラー、null/undefined 参照、未処理の例外）
  - 到達不能コードや常に真/偽となる条件分岐
  - 不整合な状態遷移（初期化前の使用、二重解放）
- [25010-信頼性: 可用性] システムが利用可能な状態を維持する能力
  - リソースリーク（未クローズのコネクション、ファイルハンドル、メモリ）
  - タイムアウト設定の欠如（外部 API 呼び出し、DB 接続）
  - 無限ループまたは非終了の再帰の可能性
- [25010-信頼性: 障害許容性] 障害発生時にも動作を継続する能力
  - エラーハンドリングの欠如（外部 API 呼び出し、ファイル I/O、ネットワーク操作）
  - 例外の握りつぶし（catch ブロックが空、エラー情報の消失）
  - フォールバック処理の不備
- [25010-信頼性: 回復性] 障害後の状態復旧能力
  - トランザクション管理の不備（部分的な更新によるデータ不整合の可能性）
  - リトライ処理の欠如（べき等でない操作のリトライは逆に指摘）

#### [25010-セキュリティ] セキュリティ — Security

> **注**: セキュリティの詳細なレビューは `/review-security`（OWASP Top 10 / CWE Top 25 準拠）を使用してください。ここでは基本的なチェックのみ行います。

**[5055-SEC] セキュリティの自動測定指標（簡易チェック）**

- [25010-セキュリティ: 機密性] ハードコードされた秘密情報（API キー、パスワード、トークン）
- [25010-セキュリティ: インテグリティ] 入力バリデーションの明らかな欠如
- [25010-セキュリティ: 責任追跡性] セキュリティイベントのログ記録の欠如

> 上記以外のセキュリティ問題が疑われる場合は `/review-security` の実行を推奨してください。

#### [25010-保守性] 保守性 — Maintainability

> ソフトウェアが修正される際の効果性と効率性

**[5055-MNT] 保守性の自動測定指標**

- [25010-保守性: モジュール性] モジュール間の結合度と凝集度
  - 過度な結合（他モジュールの内部実装への依存）
  - 循環依存
  - 神クラス / 神関数（過大な責務を持つコンポーネント）
- [25010-保守性: 再利用性] コードの再利用可能性
  - コードの重複（DRY 原則の違反）
  - ハードコードされた設定値（マジックナンバー、マジックストリング）
- [25010-保守性: 解析性] コードの理解しやすさ
  - 不整合な命名やコーディングスタイル（プロジェクトの既存規約と比較）
  - 過度に複雑な制御フロー（深いネスト、複雑な条件式）
  - [5055-MNT] 循環的複雑度が高い関数
- [25010-保守性: 修正性] 変更の容易さ
  - 単一責任原則の違反
  - 変更の影響範囲が不明確（グローバル状態への依存）
- [25010-保守性: 試験性] テストのしやすさ
  - テスト困難な設計（副作用の多い関数、依存注入の欠如）
  - 新機能に対するテストの有無
  - テストが意味のあるアサーションをしているか

#### [25010-性能効率性] 性能効率性 — Performance Efficiency

> 使用するリソースの量に対するパフォーマンスの度合い

**[5055-PER] 性能効率性の自動測定指標**

- [25010-性能効率性: 時間効率性] 応答時間とスループット
  - N+1 クエリ問題
  - ループ内での不必要な I/O 操作（DB クエリ、API 呼び出し、ファイル読み込み）
  - 不要な同期処理（非同期化可能な処理のブロッキング）
- [25010-性能効率性: 資源効率性] リソースの使用量
  - メモリ効率の悪いデータ構造やアルゴリズムの選択
  - 大量データの一括読み込み（ページネーション/ストリーミングの欠如）
  - 不要なオブジェクトの生成・コピー
- [25010-性能効率性: 容量満足性] パラメータの上限
  - バッファサイズ、キューサイズ等の制限の欠如
  - スケーラビリティを阻害するボトルネック

#### [25010-機能適合性] 機能適合性 — Functional Suitability

> ソフトウェアが明示的・暗黙的なニーズを満たす機能を提供する能力

- [25010-機能適合性: 機能完全性] 要件に対する機能の網羅性
  - 実装されるべきエッジケースの欠落
  - 境界値の処理漏れ
- [25010-機能適合性: 機能正確性] 正確な結果を提供する能力
  - 計算ロジックの誤り
  - データ型の不整合（暗黙の型変換による精度損失）
- [25010-機能適合性: 機能適切性] タスクの目的に対する適切さ
  - プロジェクトの既存アーキテクチャパターンとの不整合

#### [25010-互換性] 互換性 — Compatibility

- [25010-互換性: 共存性] 他コンポーネントとの共存
  - 共有リソースの競合（ポート、ファイル、グローバル状態）
- [25010-互換性: 相互運用性] 外部システムとのインターフェース
  - API 契約の違反（型の不一致、必須フィールドの欠落）
  - 後方互換性の破壊（公開 API の変更）

#### [25010-使用性] 使用性 — Usability

- [25010-使用性: 運用性] ユーザーが操作・制御しやすいか
  - 不明瞭なエラーメッセージ（ユーザーが対処方法を理解できない）
  - 設定やオプションの不整合
- [25010-使用性: ユーザーエラー防止] 誤操作の防止
  - 危険な操作に対する確認の欠如
  - 不可逆な操作のガード不足

#### [25010-移植性] 移植性 — Portability

- [25010-移植性: 適応性] 異なる環境への適応
  - プラットフォーム固有のパス区切り文字やコマンドのハードコード
  - 環境依存の値のハードコード（IP アドレス、ファイルパス）
- [25010-移植性: 設置性] インストール・セットアップの容易さ
  - 文書化されていない環境前提条件

### Step 4: 結果の報告

レビュー結果を以下の形式で表示する:

```
═══ コード品質レビュー結果 ═══

準拠規格: ISO/IEC 25010:2023 / ISO/IEC 5055:2021
対象: {コミット範囲} ({N} コミット, {M} ファイル変更)

## 問題なし ✓
- {チェック済みで問題のなかった品質特性と対応する規格識別子}

## 要修正（致命的）
- {ファイル}:{行} - [{識別子}] {問題の説明}

## 要修正（重要）
- {ファイル}:{行} - [{識別子}] {問題の説明}

## 提案（軽微）
- {ファイル}:{行} - [{識別子}] {改善の提案}

---
致命的: {N} / 重要: {N} / 軽微: {N}
```

### Step 5: 修正タスクリスト生成（問題がある場合）

「致命的」または「重要」の問題が 1 つ以上ある場合:

1. 修正タスクリストを生成する
2. 形式は decompose.md の出力と同じ Markdown チェックリスト
3. 各タスクには問題の場所（ファイル:行）、規格識別子、修正方針を記載

```markdown
# コード品質レビュー修正タスク

> レビュー対象: {コミット範囲}
> 準拠規格: ISO/IEC 25010:2023 / ISO/IEC 5055:2021
> 生成日: {YYYY-MM-DD}

## タスク
- [ ] {ファイル}:{行} - [{識別子}] {具体的な修正内容}
...
```

4. 出力パスを表示し、ralph-loop での実行コマンド例を案内

### Step 6: 問題がない場合

全チェックで問題が見つからなかった場合:
- 「問題は検出されませんでした」と報告
- 修正タスクリストは生成しない

## ルール

- **変更のコンテキストを理解する**: 差分だけでなく、変更後のファイル全体を読む
- **プロジェクト規約を基準にする**: 一般的なベストプラクティスよりも、プロジェクトの既存パターンを優先する
- **誤検出を避ける**: 確信のない問題は「提案（軽微）」として報告し、致命的・重要には含めない
- **修正可能な問題のみ報告**: 設計方針への意見や好みの問題は報告しない
- **セキュリティの深掘りは `/review-security` へ**: セキュリティ上の重大な懸念が見つかった場合は `/review-security` の実行を推奨する
- **言語・フレームワークに応じた判断**: 使用言語やフレームワークが提供する品質機能を考慮し、過剰な指摘を避ける
